Unit 1 — Introduction & Architecture

IoT Definition, Characteristics and Components
+
Definition
Internet of Things is a network of physical objects embedded with sensors, software, and connectivity to exchange data with other devices and systems over the internet. It enables smart decision-making through real-time data collection and analysis.
Key Points
  • Characteristics: Connectivity, Intelligence, Dynamic Nature, Enormous Scale, Sensing, Heterogeneity
  • Components: Sensors/Actuators, Connectivity, Data Processing, User Interface
  • Sensors collect data from environment; Actuators perform actions based on commands
  • Connectivity protocols: WiFi, Bluetooth, Zigbee, LoRaWAN, Cellular
  • Edge computing processes data locally; Cloud handles heavy analytics
  • User interfaces include mobile apps, web dashboards, voice assistants

IoT Reference Architecture (Physical and Logical Design)
+
Definition
IoT Reference Architecture provides a standardized framework for designing IoT systems. Physical design focuses on hardware components and their connections, while logical design defines functional layers and data flow.
Key Points
  • Physical Design: Things, Connectivity, Edge Computing, Data Accumulation, Data Abstraction, Application
  • Logical Design: Perception Layer, Network Layer, Middleware Layer, Application Layer
  • Perception Layer contains sensors and actuators for data collection
  • Network Layer handles communication protocols and data transmission
  • Middleware Layer provides data processing, storage, and device management
  • Application Layer delivers services to end users and business logic

IoT Framework and Communications Model
+
Definition
IoT Framework defines the structure for building IoT solutions including hardware, software, and communication protocols. Communications Model specifies how devices exchange data using request-response, publish-subscribe, push-pull, and exclusive pair patterns.
Key Points
  • Framework components: Device, Communication, Services, Management, Security, Application
  • Request-Response: Client sends request, server responds (HTTP, CoAP)
  • Publish-Subscribe: Publishers send data to topics, subscribers receive (MQTT)
  • Push-Pull: Producers push data to queue, consumers pull when ready
  • Exclusive Pair: Bidirectional connection between two endpoints (WebSocket)
  • Communication protocols: MQTT, CoAP, AMQP, HTTP, WebSocket, DDS

IoT vs M2M vs WoT
+
Definition
IoT connects devices via internet for data exchange and automation. M2M is direct device-to-device communication without internet. WoT applies web standards to IoT for interoperability using HTTP, REST, and web technologies.
Key Points
  • IoT: Internet-based, cloud-centric, scalable, uses IP protocols, open standards
  • M2M: Point-to-point, proprietary protocols, limited scalability, closed systems
  • WoT: Web-based IoT, uses HTTP/REST, JSON/XML, web APIs, browser-accessible
  • IoT focuses on connectivity and data; WoT focuses on interoperability
  • M2M examples: ATM networks, vending machines; IoT: smart homes, cities
  • WoT enables web browsers to directly interact with physical devices

IoT Gateway and Proxy
+
Definition
IoT Gateway is a physical device that connects sensors to the cloud, performing protocol translation, data filtering, and edge processing. Proxy acts as an intermediary for communication, providing security, caching, and load balancing between devices and servers.
Key Points
  • Gateway functions: Protocol translation, data aggregation, edge analytics, security
  • Gateway connects heterogeneous devices using different protocols to cloud
  • Proxy types: Forward proxy (client-side), Reverse proxy (server-side)
  • Proxy functions: Security filtering, caching, load balancing, anonymity
  • Gateway performs local processing to reduce cloud traffic and latency
  • Both provide security through encryption, authentication, and access control

Unit 2 — Sensing & Hardware

Sensor Node Architecture, Classes and Error Types
+
Definition
Sensor node is a self-contained device with sensing, processing, communication, and power components that collects environmental data. Sensor classes categorize sensors by measurement type, and errors occur due to calibration, environmental factors, or hardware limitations.
Key Points
  • Architecture: Sensing Unit, Processing Unit, Transceiver, Power Unit, Memory
  • Classes: Analog (continuous output), Digital (discrete output), Passive (no power), Active (requires power)
  • Error types: Bias (constant offset), Drift (slow change over time), Hysteresis (path-dependent), Quantization (ADC rounding)
  • Processing unit includes microcontroller and memory for local computation
  • Transceiver enables wireless communication using protocols like Zigbee, BLE
  • Power management critical for battery-operated nodes using sleep modes

Actuator Types and Applications
+
Definition
Actuators are devices that convert electrical signals into physical actions like motion, heat, or light. They enable IoT systems to interact with the physical world based on sensor data and control commands.
Key Points

ZigBee Architecture and Applications
+
Definition
ZigBee is a low-power, low-data-rate wireless communication protocol based on IEEE 802.15.4 standard designed for IoT applications. It operates in 2.4 GHz with mesh networking capability for reliable device-to-device communication.
Key Points
  • Stack layers: Physical (IEEE 802.15.4), MAC, Network, Application Support Sublayer, Application
  • Device types: Coordinator (network manager), Router (routes packets), End Device (sensor/actuator)
  • Topologies: Star, Tree, Mesh (self-healing, multi-hop routing)
  • Features: Ultra-low power (battery life in years), 250 kbps, 10-100m range, 65000 nodes
  • Security: 128-bit AES encryption, authentication, access control
  • Applications: Home automation, industrial monitoring, healthcare, smart lighting, building automation
Diagram
ZigBee Stack: Application Layer (ZCL, Profiles) ↓ Application Support Sublayer (APS) ↓ Network Layer (Routing, Security) ↓ MAC Layer (CSMA/CA) ↓ Physical Layer (IEEE 802.15.4) Topologies: STAR: TREE: MESH: [C] [C] [C]--[R] /|\ | | X | [E][E][E] [R][R] [R]--[R] /\ | | | [E][E][E] [E] [E] C=Coordinator R=Router E=End Device

Bluetooth vs ZigBee
+
Definition
Bluetooth is a short-range wireless technology for high-speed data transfer between consumer devices. ZigBee is designed for low-power, low-data-rate IoT sensor networks with mesh topology and years-long battery life.
Key Points

  • ZigBee better for large-scale sensor networks; Bluetooth better for audio and peripherals
  • Both operate at 2.4 GHz with 128-bit security

IEEE 802.15.4 Standard
+
Definition
IEEE 802.15.4 is a technical standard defining Physical and MAC layers for low-rate wireless personal area networks (LR-WPAN). It is the foundation for ZigBee, 6LoWPAN, and other IoT protocols requiring low power and low data rate communication.
Key Points
  • Defines only Physical and MAC layers; upper layers defined by ZigBee, 6LoWPAN etc.
  • Frequency bands: 868 MHz (Europe, 1 ch), 915 MHz (Americas, 10 ch), 2.4 GHz (Global, 16 ch)
  • Data rates: 20 kbps (868 MHz), 40 kbps (915 MHz), 250 kbps (2.4 GHz)
  • Full Function Device (FFD): can be coordinator or router; Reduced Function Device (RFD): end device only
  • MAC features: CSMA/CA channel access, beacon and non-beacon modes, GTS (guaranteed time slots)
  • Frame types: Data, Acknowledgment, Beacon, MAC Command
Diagram
Stack Position: Upper Layers (ZigBee, 6LoWPAN, Thread...) ───────────────────────────────────────── IEEE 802.15.4 Standard: MAC Layer (CSMA/CA, Framing, ACK) ↓ Physical Layer (RF Transceiver) ───────────────────────────────────────── Frequency Bands: 868 MHz → 1 channel, 20 kbps (Europe) 915 MHz → 10 channels, 40 kbps (Americas) 2.4 GHz → 16 channels, 250 kbps (Global) Device Types: FFD ─┬─ Coordinator (1 per network) └─ Router / End Device RFD ─── End Device only (battery sensors)

RFID Working Principle and Applications
+
Definition
Radio Frequency Identification uses electromagnetic fields to automatically identify and track tags attached to objects. RFID system consists of tags (transponders) storing data and readers (interrogators) that wirelessly communicate using radio waves.
Key Points
  • Components: RFID Tag (chip + antenna), RFID Reader (transceiver), Backend System (database)
  • Tag types: Passive (no battery, powered by reader field), Active (battery-powered, long range), Semi-passive (battery for chip, reader powers comms)
  • Working: Reader emits RF → Tag receives energy → Tag transmits ID → Reader decodes → Backend processes
  • Frequency bands: LF 125 kHz, HF 13.56 MHz, UHF 860-960 MHz, Microwave 2.45 GHz
  • Passive range: 10 cm to 10 m; Active range: up to 100 m
  • Applications: Supply chain, inventory, access control, payments, livestock, toll collection
Diagram
RFID System Architecture: [RFID Reader] ←──RF Signal──→ [RFID Tag] ↓ (Chip + Antenna) [Middleware] ↓ [Backend Database] Working (5 steps): 1. Reader emits RF electromagnetic field 2. Tag antenna captures energy from field 3. Tag chip powers up and encodes ID 4. Tag modulates and reflects signal back 5. Reader decodes → sends to backend DB Tag Comparison: PASSIVE: No battery │ Short range │ Cheap │ LF/HF ACTIVE: Battery │ Long range │ Cost++ │ UHF SEMI-PASSIVE:Battery for chip, reader power comms

6LoWPAN and Wireless Sensor Network
+
Definition
6LoWPAN (IPv6 over Low-Power Wireless Personal Area Networks) enables IPv6 packets over IEEE 802.15.4 networks via an adaptation layer. WSN is a network of spatially distributed autonomous sensors monitoring physical conditions and passing data cooperatively to a gateway.
Key Points
  • 6LoWPAN adaptation layer: header compression, fragmentation, IPv6 addressing, mesh routing
  • Sits between the IPv6 Network Layer and the IEEE 802.15.4 MAC Layer
  • WSN architecture: Sensor nodes → Sink/Gateway → Internet → End user/Task manager
  • WSN characteristics: Self-organizing, multi-hop routing, energy-constrained, fault-tolerant, scalable
  • WSN routing protocols: LEACH (clustering), RPL (IPv6 routing), AODV (on-demand)
  • Applications: Environmental monitoring, precision agriculture, structural health, military surveillance
Diagram
6LoWPAN Protocol Stack: Application Layer ↓ Transport Layer (UDP/TCP) ↓ IPv6 Network Layer ↓ ════════════════════════════════ 6LoWPAN Adaptation Layer (Header Compression, Fragmentation) ════════════════════════════════ ↓ IEEE 802.15.4 MAC ↓ IEEE 802.15.4 PHY WSN Architecture: [Sensor]─┐ [Sensor]─┼──→ [Sink/Gateway] ──→ [Internet] ──→ [User] [Sensor]─┘ Multi-hop mesh routing between nodes

Unit 3 — Communication Protocols

MQTT Architecture and Communication
+
Definition
MQTT (Message Queuing Telemetry Transport) is a lightweight publish-subscribe messaging protocol for constrained devices and low-bandwidth networks. A broker-based architecture decouples publishers and subscribers through topic-based message channels.
Key Points
  • Architecture: Publishers, Broker (server), Subscribers, Topics (message channels)
  • QoS 0: at most once (fire and forget); QoS 1: at least once (ACK); QoS 2: exactly once (4-way handshake)
  • Features: Lightweight, low bandwidth, persistent sessions, Last Will Testament, retained messages
  • Message types: CONNECT, CONNACK, PUBLISH, SUBSCRIBE, SUBACK, UNSUBSCRIBE, PINGREQ, DISCONNECT
  • Topic hierarchy uses / separator — example: home/bedroom/temperature
  • Ports: 1883 (unencrypted), 8883 (TLS/SSL)
Diagram
MQTT Architecture: [Publisher 1] ──PUBLISH──→ [MQTT BROKER] ──PUBLISH──→ [Subscriber 1] [Publisher 2] ──PUBLISH──→ (Topics A,B) ──PUBLISH──→ [Subscriber 2] Communication Flow: 1. Client connects to broker (CONNECT / CONNACK) 2. Subscriber subscribes (SUBSCRIBE / SUBACK) 3. Publisher publishes message (PUBLISH) 4. Broker forwards to all subs (PUBLISH) 5. Client disconnects (DISCONNECT) QoS Levels: QoS 0: Publisher → Broker (no ACK, may lose) QoS 1: Publisher → Broker → ACK (may duplicate) QoS 2: 4-step handshake (guaranteed exactly once)

CoAP Protocol and Message Types
+
Definition
CoAP (Constrained Application Protocol) is a specialized RESTful web protocol for constrained IoT devices and lossy networks. It uses UDP instead of TCP for low overhead, targeting devices with limited memory and processing power.
Key Points
  • Message types: Confirmable (CON) — needs ACK; Non-confirmable (NON) — no ACK; Acknowledgement (ACK); Reset (RST)
  • Methods: GET (read), POST (create), PUT (update), DELETE (remove)
  • Features: UDP-based, low overhead, observe pattern, multicast support, asynchronous
  • Message format: 4-byte fixed header + Token + Options + Payload
  • Response codes: 2.xx Success, 4.xx Client Error, 5.xx Server Error
  • Ports: 5683 (default UDP), 5684 (DTLS secured)
Diagram
CoAP Message Types: CONFIRMABLE (CON) — reliable: Client ──CON──→ Server Client ←─ACK──┘ NON-CONFIRMABLE (NON) — unreliable: Client ──NON──→ Server (No acknowledgement sent) CoAP Message Format: ┌─────────────────────────────┐ │ Ver │ T │ TKL │ Code │ MsgID│ ← 4 bytes header ├─────────────────────────────┤ │ Token (0-8 bytes) │ ├─────────────────────────────┤ │ Options (variable) │ ├─────────────────────────────┤ │ Payload (variable) │ └─────────────────────────────┘ Example: Request: GET coap://sensor.local/temperature (CON) Response: 2.05 Content → "25.4°C" (ACK)

AMQP Features and Components
+
Definition
AMQP (Advanced Message Queuing Protocol) is an open standard application layer protocol for enterprise message-oriented middleware. It provides reliable message delivery with queuing, routing, transactions, and security for enterprise-grade IoT applications.
Key Points
  • Components: Producer (sender), Exchange (router), Binding (rules), Queue (buffer), Consumer (receiver), Broker
  • Exchange types: Direct (exact routing key), Fanout (broadcast to all), Topic (wildcard pattern), Headers (attribute match)
  • Features: Reliable delivery, acknowledgment, transactions, SASL/TLS security, flow control, persistence
  • Delivery modes: Persistent (disk, survives restart), Non-persistent (memory only)
  • Message properties: Priority, timestamp, expiry, content-type, correlation-id, reply-to
  • Ports: 5672 (default), 5671 (TLS/SSL)
Diagram
AMQP Architecture: [Producer] → [Exchange] → [Queue 1] → [Consumer 1] ↓ [Queue 2] → [Consumer 2] Exchange Types: DIRECT : Producer → Exchange ──(key match)──→ Queue FANOUT : Producer → Exchange ──(broadcast)──→ All Queues TOPIC : Producer → Exchange ──(*.error)────→ Matching Queues HEADERS: Producer → Exchange ──(attr match)──→ Queue Message Flow: 1. Producer publishes message to Exchange 2. Exchange applies routing rules (bindings) 3. Message placed into matching Queue(s) 4. Consumer retrieves and sends ACK 5. Broker removes message after ACK

XMPP and SMQTT
+
Definition
XMPP (Extensible Messaging and Presence Protocol) is an XML-based open standard for real-time messaging and presence information. SMQTT (Secure MQTT) is MQTT with TLS/SSL encryption providing certificate-based authentication for secure IoT communication.
Key Points
  • XMPP features: Real-time messaging, presence information, XML stanzas, extensible, decentralized federation
  • XMPP addressing: user@domain/resource format (e.g. sensor1@home.local/living-room)
  • XMPP stanza types: Message (content), Presence (online/offline status), IQ (info query/response)
  • SMQTT: standard MQTT over TLS/SSL on port 8883 with X.509 certificate authentication
  • SMQTT security layers: TLS handshake, certificate verification, encrypted data channel, ACL authorization
  • XMPP suited for human-to-human and device-to-human; SMQTT for machine-to-machine IoT

REST vs SOAP vs CoAP
+
Definition
REST is a stateless architectural style using HTTP for web service APIs. SOAP is a strict XML-based messaging protocol with built-in error handling and security for enterprise systems. CoAP is a UDP-based RESTful protocol designed specifically for constrained IoT devices.
Key Points
  • REST: HTTP-based, stateless, JSON/XML, GET/POST/PUT/DELETE, lightweight, cacheable, easy to implement
  • SOAP: XML envelope format, stateful, WSDL service contracts, WS-Security, heavyweight, enterprise-grade reliability
  • CoAP: UDP-based, REST-like methods, binary encoding, observe pattern, multicast, ultra-lightweight for IoT
  • REST for web APIs and mobile backends; SOAP for enterprise integration and banking; CoAP for constrained sensors
  • REST uses HTTP status codes; SOAP uses Fault elements; CoAP uses numeric response codes (2.05, 4.04)
  • Performance order: CoAP (fastest) > REST (moderate) > SOAP (slowest due to XML overhead)
Diagram
Comparison Table: Feature | REST | SOAP | CoAP -------------|----------------|----------------|------------------ Protocol | HTTP | HTTP / SMTP | UDP Data Format | JSON / XML | XML only | Binary (compact) Transport | TCP | TCP | UDP Overhead | Low | High (XML) | Very Low State | Stateless | Stateful | Stateless Methods | GET/POST/PUT | Custom Ops | GET/POST/PUT/DEL Security | HTTPS (TLS) | WS-Security | DTLS Error Handle | HTTP codes | Fault elements | CoAP codes Use Case | Web APIs | Enterprise/Bank| IoT Sensors Performance | Moderate | Slow | Fast Request Examples: REST: GET /api/temperature HTTP/1.1 SOAP: <soap:Envelope><Body><GetTemp/></Body></soap:Envelope> CoAP: GET coap://device/temp (binary, 4-byte header)

+
Definition
Cloud computing for IoT provides scalable on-demand infrastructure for data storage, processing, and analytics. It offers storage models (file, block, object), communication APIs, and service tiers (IaaS, PaaS, SaaS) enabling IoT without managing physical servers.

IoT Security Challenges, Attacks and Vulnerability Analysis
+
Definition
IoT security protects connected devices, networks, and data from unauthorized access and malicious attacks. The massive scale, resource constraints, and heterogeneity of IoT create unique security challenges requiring layered defense strategies.
Key Points
  • Challenges: Limited compute/memory for encryption, no standardization, weak default authentication, no update mechanisms, physical exposure
  • Physical attacks: device tampering, hardware cloning, side-channel analysis, theft
  • Network attacks: eavesdropping, Man-in-the-Middle (MITM), DDoS, replay attacks, jamming
  • Software attacks: malware injection, firmware exploits, backdoors, command injection
  • Famous attack — Mirai Botnet (2016): infected 600,000 IoT cameras/routers → largest DDoS in history
  • Mitigation: TLS/DTLS encryption, certificate auth, secure boot, OTA updates, network segmentation, zero-trust
Diagram
IoT Attack Surface (3 layers): PHYSICAL LAYER: [Device] ←── Tampering, Cloning, Side-channel NETWORK LAYER: [Channel] ←── Eavesdrop, MITM, DDoS, Replay, Jamming APPLICATION LAYER: [Software] ←── Malware, Injection, Backdoor, Exploit Layered Security Mitigation: ───────────────────────────────────────── Application : Secure coding, Input validation Data : Encryption at rest and in transit Network : Firewall, IDS/IPS, VPN, Segmentation Device : Auth, Secure boot, TPM, OTA updates Physical : Tamper detection, Access control ───────────────────────────────────────── Real Attacks: Mirai Botnet → DDoS via 600K infected IoT devices MITM Attack → Intercept and alter device commands Replay Attack → Re-send captured valid packets Firmware Hack → Flash malicious firmware over OTA

Data Analytics and Data Store in IoT
+
Definition
IoT data analytics extracts actionable insights from continuous sensor streams using statistical, machine learning, and AI techniques. Data stores manage high-volume, high-velocity IoT data using specialized databases optimized for time-series, scale, and query patterns.

IoT Design Methodology and ITU-IoT Reference Model
+
Definition
IoT Design Methodology is a structured 6-step process from requirements to deployment. The ITU-T Y.2060 IoT Reference Model standardizes a 4-layer architecture (Application, Service Support, Network, Device) used globally for IoT system design.
Key Points
  • Design steps: (1) Purpose and Requirements, (2) Process Specification, (3) Domain Model, (4) Information Model, (5) Service Specifications, (6) IoT Level Specification
  • ITU-T layers: Application (user interfaces/apps), Service Support and Application Support (middleware/APIs), Network (connectivity), Device (sensors/actuators/gateways)
  • Cross-cutting capabilities: Security, Management, QoS (span all 4 layers)
  • IoT levels 1-6 define complexity: L1=single node, L2=local, L3=cloud, L4=multi-cloud, L5=collaborative, L6=central control
  • Domain model defines entities, relationships, and attributes of the IoT system
  • Design considerations: scalability, interoperability, security, power, cost, reliability

NFV, SDN and NFC
+
Definition
NFV (Network Function Virtualization) runs network functions as software on standard servers. SDN (Software Defined Networking) separates control and data planes for programmable networks. NFC (Near Field Communication) enables contactless data exchange within 4 cm range.
Key Points
  • NFV: replaces dedicated hardware (routers, firewalls, load balancers) with VMs on standard x86 servers — reduces cost and increases flexibility
  • SDN: centralized controller manages network via OpenFlow protocol; separates control plane (decisions) from data plane (forwarding)
  • SDN layers: Application Layer (business logic), Control Layer (SDN Controller = brain), Infrastructure Layer (dumb switches/routers)
  • NFC: 13.56 MHz, 4 cm range, 106-424 kbps; modes: Reader/Writer, Peer-to-Peer, Card Emulation
  • NFC applications: contactless payments, access cards, device pairing, smart posters, ticketing
  • IoT use: NFV/SDN provide flexible network slicing; NFC used for device provisioning and secure pairing

Quick Reference — Other Topics

ICT Trends and Megatrends in IoT
ICT trends (cloud, big data, AI/ML, 5G) provide the technological backbone enabling IoT at scale. Megatrends (urbanization, aging population, climate change) create the societal demand driving IoT adoption worldwide.
Like how roads and electricity had to exist before cars became useful — ICT trends are the infrastructure that make IoT solutions viable.
  • 5G enables massive IoT with ultra-low latency (1 ms) and 1 million devices per km²
  • AI/ML converts raw sensor data into intelligent decisions — the brain behind smart IoT systems
  • Urbanization drives smart city demand: 68% of world population in cities by 2050 needs smart infrastructure
IoT Applications (Emerging)
Emerging IoT applications span smart cities, precision healthcare, autonomous vehicles, connected retail, energy grids, and environmental monitoring — transforming every industry through data-driven automation.
Just as electricity moved from powering factories to every device in your home, IoT is expanding from industrial monitoring into every aspect of daily life.
  • Smart Cities: intelligent traffic, waste bin sensors, smart parking, air quality monitoring
  • Connected Healthcare: wearable monitors, remote patient care, smart pill dispensers, asset tracking in hospitals
  • Precision Agriculture: drone-based crop analysis, automated irrigation, livestock health wearables
Participatory Sensing
Participatory sensing leverages smartphones and wearables carried by ordinary citizens to collectively gather large-scale environmental and social data. Citizens act as mobile sensing nodes contributing to distributed monitoring networks.
Like traffic apps that use your phone's GPS to calculate congestion for everyone — participatory sensing uses millions of citizen phones to map environmental conditions no single sensor grid could cover.
  • Uses built-in smartphone sensors: GPS, microphone, camera, accelerometer, barometer
  • Applications: urban noise mapping, air quality crowdsourcing, pothole detection, disease outbreak tracking
  • Challenge: privacy concerns, data quality variation, battery consumption, participation incentives
IP Addressing in IoT
IoT requires unique IP addresses for every device. IPv4 (32-bit, 4.3 billion addresses) is exhausted. IPv6 (128-bit) provides 340 undecillion addresses — enough for every IoT device conceivable.
IPv4 is like a city running out of phone numbers — IPv6 gives every grain of sand on Earth its own unique address, solving the IoT scale problem permanently.
  • IPv6 = 128-bit addresses → 3.4 × 10³⁸ unique addresses — every sensor can have a globally routable address
  • IPv6 supports auto-configuration (SLAAC) — IoT devices configure their own addresses without DHCP
  • 6LoWPAN compresses IPv6 headers from 40 bytes to as few as 2 bytes for constrained IoT devices
Web Sockets
WebSocket is an application layer protocol providing persistent, full-duplex bidirectional communication over a single TCP connection. It upgrades from HTTP and maintains an open channel eliminating repeated polling for live IoT dashboards.
HTTP is like exchanging letters — you send one, wait for a reply. WebSocket is like a phone call that stays open so both sides can speak the moment something happens.
  • Starts with HTTP Upgrade handshake, then switches to persistent WebSocket frame protocol
  • Reduces overhead dramatically vs HTTP polling — ideal for real-time sensor dashboards and live alerts
  • Supports both text (JSON) and binary frames; used in IoT monitoring UIs, chat, gaming, live tracking
Information Exchange Patterns in IoT
Information exchange patterns define the structural models for how IoT components communicate data: request-response, publish-subscribe, push-pull, and exclusive pair — each suited to different latency, reliability, and scalability needs.
Like different postal services: registered mail (request-response), newsletter subscription (pub-sub), PO box pickup (push-pull), and a direct phone call (exclusive pair).
  • Request-Response: synchronous, client-initiated — used in HTTP and CoAP for on-demand sensor reads
  • Publish-Subscribe: asynchronous, decoupled through broker — used in MQTT for continuous sensor streams
  • Push-Pull: queue-buffered, load-balanced — used in AMQP; producer and consumer operate at different speeds
IoT Functional View
The IoT Functional View organizes system capabilities into logical functional groups — device, communication, service, data management, security, and management — independent of physical implementation or deployment topology.
Like a restaurant's departments (kitchen, service, billing, security) that all work together regardless of the building's floor plan — functional groups define what the system does, not how it is physically wired.
  • Device Functional Group: sensor/actuator lifecycle, firmware updates, device registry
  • Communication Functional Group: protocol management, message routing, QoS, data encoding
  • Service Functional Group: middleware APIs, application enablement, data processing, event management
OGC Functional Architecture
The Open Geospatial Consortium (OGC) Sensor Web Enablement (SWE) framework defines open standards so any geospatial IoT sensor can interoperate with any application. Core services: SOS, SPS, and SAS.
Like GPS standards that allow any map app on any phone from any brand to give you accurate navigation — OGC standards ensure any sensor feeds any geospatial platform without custom integration.
  • SOS (Sensor Observation Service): query and retrieve observations from sensors in standard format
  • SPS (Sensor Planning Service): remotely task and configure sensors to collect specific data
  • SAS (Sensor Alert Service): subscribe to notifications when sensor readings cross defined thresholds
IETF Working Groups
The Internet Engineering Task Force (IETF) develops IoT networking standards through specialized working groups. Key groups: 6LoWPAN (IPv6 on low-power networks), CoRE (constrained RESTful environments), and ROLL (routing in lossy networks).
Like the Bureau of Standards that writes universal rules so plugs from any manufacturer fit any socket — IETF writes the internet rules so any IoT device interoperates with any network worldwide.
  • 6LoWPAN WG: IPv6 header compression and adaptation for IEEE 802.15.4 (foundation of modern IoT networking)
  • CoRE WG: CoAP protocol, resource discovery (/.well-known/core), and RESTful IoT API design
  • ROLL WG: RPL (Routing Protocol for LLNs) — IPv6 routing optimized for low-power lossy sensor networks
Service Choreography in IoT
Service choreography is decentralized coordination where multiple IoT services collaborate autonomously based on shared events and messages. No central orchestrator — each service independently knows when and how to respond.
Like a jazz band with no conductor — each musician listens to the others and responds instinctively, creating coordinated music without anyone directing. Each IoT service responds to events without a central controller.
  • Decentralized: services are self-coordinating; failure of one service does not stop others
  • Event-driven: services trigger on messages, sensor readings, or state changes — not on explicit commands
  • Contrasts with orchestration (central engine directs all): choreography is more resilient but harder to debug
IoT Deployment and Operational View
Deployment view maps the physical distribution of IoT system components across edge, fog, and cloud tiers. Operational view covers the ongoing lifecycle: device provisioning, monitoring, software updates, fault management, and decommissioning.
Like a city's water supply — deployment view is the map of pipes, pumps, and reservoirs; operational view is the daily monitoring schedule, maintenance crew, and emergency response plan.
  • Edge tier: sensors/actuators at the data source — lowest latency, limited processing
  • Fog tier: intermediate gateway/local server — pre-processes data to reduce cloud bandwidth
  • Cloud tier: centralized analytics, storage, device management, and application hosting for entire fleet
IoT Privacy and Control Solutions
IoT privacy protects personal data collected by connected devices through technical and policy measures. Control solutions give users transparency, consent management, and the ability to access, modify, or delete their data.
Like a lockable diary with a permission slip — privacy is the lock that keeps data secure; control solutions are your power to choose who gets a key and for how long.
  • Data minimization: collect only the minimum data necessary — do not store what you do not need
  • Consent management: explicit user approval before data collection; opt-in preferred over opt-out
  • Privacy-by-design: security and anonymization built into architecture from day one, not bolted on later
Smart Street Light System
Smart street lighting uses IoT sensors and network connectivity to automatically adjust brightness based on real-time conditions — ambient light levels, pedestrian motion, and time of day — reducing energy consumption by up to 70%.
Like a well-trained security guard who dims the lights in an empty corridor but brightens them the instant someone approaches — automated, energy-aware, and always on.
  • Sensors used: LDR (ambient light), PIR (motion detection), current/voltage sensors (fault detection), environmental (air quality)
  • Communication: ZigBee or cellular mesh network connecting each pole to a central management system
  • Benefits: 50-70% energy savings, remote fault diagnosis, reduced maintenance visits, integration with emergency services
